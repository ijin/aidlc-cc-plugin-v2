# Working in this repo (agent guide)

This repo packages the **v2** rewrite of [awslabs/aidlc-workflows](https://github.com/awslabs/aidlc-workflows)
as a Claude Code plugin (`aidlc-v2`). Read `README.md` for the full picture; this file is the short
list of **rules and invariants** — break one and you break the plugin or its provenance.

This file is **committed and public** — keep it environment-neutral (no machine paths, tooling, or
provider names). Any *local* workflow notes (how a particular machine runs the review/smoke) go in
`CLAUDE.local.md`, which is gitignored.

## Layout & the build model (internalize this first)

- `src/` is a **pristine, vendored mirror** of upstream's built `dist/claude` (its own Claude Code
  target) at the commit pinned in `UPSTREAM.lock` — normally a `v2.x` release-tag commit on
  upstream's `v2` branch. **NEVER hand-edit `src/`.** It is replaced wholesale by
  `sync-upstream.sh`; a manual edit is detected and rejected (the skew guard / dist-freshness check).
- `dist/claude/` is the **built, committed plugin** — the *only* thing that ships (marketplace
  `source: ./dist/claude`). It is generated by `targets/claude/build.mjs` and contains:
  `framework/` (= `src/` **verbatim**, byte-identity is a build postcondition), the authored entry
  skill + installer (from `targets/claude/plugin/`), and the generated plugin manifest. **NEVER
  hand-edit `dist/`** — rebuild instead. `src/`, `targets/`, `test/`, `.claude/`, `.design/` do NOT ship.
- This is an **installer plugin**: upstream's engine must live at `<project>/.claude/` (its
  hooks/tools resolve paths under the project root; its `doctor` prescribes the copy). We therefore
  ship upstream's tree untouched and install it — **never patch upstream code to run in place**.
  All plugin engineering lives in **`targets/claude/`** — not in `src/`.
- After any change that feeds the build, **rebuild and commit `dist/` in the same change**
  (`node targets/claude/build.mjs && git add dist/`). A pre-commit hook + CI enforce that committed
  `dist/` equals a fresh build; `git config core.hooksPath .githooks` to enable the hook locally.
- When staging a fresh sync, use **`git add --force -A src`**: the vendored `src/.gitignore`
  matches files upstream ships force-added (e.g. `aidlc/active-space`); a plain `git add` silently
  drops them (gitignore only affects untracked files, so this matters on first introduction).

## Syncing from upstream

- To adopt a new upstream snapshot, use **`./targets/claude/sync-upstream.sh <sha>`** (or the
  guided `release-upstream` skill). It is the only correct path: it sparse-checkouts upstream's
  `dist/claude`, rebuilds, and rewrites the lock on a clean build — and never auto-commits.
- **Pin upstream release-tag commits** (peel the tag: `vX.Y.Z^{}`), not the `v2` branch tip — the
  tip is unreleased and the branch has been force-pushed before. Upstream's `main` is the v1 line.
- **Do NOT run `git subtree pull`.** `src/` is not a subtree; that command would map upstream's
  repo root into `src/` and corrupt it.
- The decision to adopt a snapshot is human; only the mechanics are automated. The **plugin version
  mirrors the adopted framework version** (`2.1.4`, or `2.1.4-pN` for plugin-only patches) — the
  build enforces it against `package.json` + `marketplace.json`.

## Verification (run before claiming done)

- `npm test` runs the free deterministic gates: contract drift-injection meta-tests, T1 triage
  meta-tests, the **installer→upstream-doctor end-to-end** (needs `bun`; skips without it), and the
  `dist/`-freshness guard. **Always green before committing.**
- **T2a (load smoke) is opt-in** — `npm run smoke` makes one billable LLM call and asserts the
  plugin loads and exposes exactly the installer surface (entry skill present, no framework leak).
  Run it when T1's `smoke.advised` fires (engine control surface changed) or before a release.
- **If the build fails on a contract check**, it names what upstream changed; the fix is in
  `targets/claude/` (e.g. new root entry → `REQUIRED_SRC_ROOT` + installer placement; new
  settings.json key → `SETTINGS_KNOWN_KEYS` + `mergeSettings()`; hook shape → `HOOK_CMD_RE`; new
  MCP server → `MCP_KNOWN_SERVERS` + README). Fix the adapter, never `src/` or `dist/`. Full table:
  `MAINTAINERS.md`.

## How to make changes here

- **Independent review is part of "done."** For any substantive change (build/installer logic, the
  contract, the verification tiers, the sync flow): make the change → get an independent adversarial
  review → reach explicit agreement → *then* it's done. Do not treat "committed" as "finished" before
  the review loop closes.
- Ship upstream content **verbatim** — if a change seems to require editing the payload, the answer
  is an installer/docs change or an upstream contribution, never a content patch.
- Validate YAML with `actionlint`; no trailing whitespace on blank lines.

## Public-repo hygiene

- This repo is intended to go public. Commit messages are **plugin/snapshot-focused, for an outside
  reader** — do NOT use internal process/tooling framing in public commit messages. The
  `release-upstream` skill already produces clean, snapshot-focused text.
- `src/` is vendored from upstream (MIT-0, Copyright Amazon); keep the README's License & attribution
  section accurate. This is an independent community port, not affiliated with AWS.
